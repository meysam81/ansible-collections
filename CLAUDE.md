# ansible-collections

Public, Apache-2.0 licensed Ansible collection (`meysam81.general`) of independently deployable, single-purpose roles. Consumed by Galaxy and by anyone who wants individual building blocks.

## ⛔ HARD RULES — NEVER VIOLATE

These rules exist because this repository is **public and infrastructure-agnostic**. Any leak of caller-specific context is a regression.

1. **No infra-specific identifiers.** Never reference any caller's domains, hostnames, IPs, ASNs, cloud accounts, S3 buckets, SSM paths, KMS keys, mailbox addresses, or person names anywhere in this repo — including inline comments, docstrings, examples, README snippets, defaults, and tests. If you need an example, use IANA-reserved names: `example.com`, `mx.example.com`, `192.0.2.0/24`, `2001:db8::/32`.

2. **No conversation refs.** Never reference chat sessions, ticket IDs, "Q1/Q2/the user said", commit-message-style "as discussed", or any artifact from a private workspace. Code and comments must read as if written for a stranger reading the repo cold.

3. **Each role stands alone.** Roles MUST work when consumed in isolation via Galaxy. No cross-role assumptions, no implicit dependency on a sibling role's state, no shared variables that aren't documented in the consuming role's `defaults/main.yml`. If two roles genuinely need to talk, that contract is documented in both READMEs and validated via molecule.

4. **Safe-by-default.** Defaults must be production-grade for the role's typical use case: TLS roles ship modern cipher lists, firewall roles ship deny-by-default, security headers are on, weak protocols are off. Loosening for compat is the caller's explicit decision via overrides — never the role's default.

5. **No required secrets in defaults.** Defaults that require a secret (API token, password, key) must be `""` and the README must document the override. Roles must never fail closed in a way that exposes credentials in logs or errors. Use `no_log: true` on tasks handling secrets.

6. **Docs live in-repo.** Every role has its own `HEADER.md` describing purpose, install, usage, and full variable table. The per-role `README.md` is always auto-generated by ansible-doctor and should NEVER be modified and all the changes should be placed in the HEADER.md instead — keep that pattern when adding variables (TOC entry + `### name` section + `Default value` block, alphabetical).

7. **Caller passes context via vars.** Anything specific to a deployment (hostnames, IP allowlists, cert paths, regions, account IDs) must be parameterised through role variables, then supplied at the call site via inventory hostvars/groupvars/inline vars in the consumer's playbook. The role itself is generic.

8. **Consumer-side context goes elsewhere.** Caller-specific overrides, secrets, ops runbooks, and infra notes belong in the *consuming* repository (e.g. an internal `infra` repo), not here. If you find yourself wanting to add an "in our deployment we…" comment to this repo, you're in the wrong file — write it in the consumer.

## Repository layout

- `roles/` — one directory per role; each is independently consumable.
- `playbook.yml` — generic test harness exercising every role; uses `lookup('env', '…')` for any caller-specific values, never hardcoded.
- `Vagrantfile` + `vagrant.yml` — local dev VM for smoke-testing roles; uses upstream Ubuntu boxes only.
- `molecule/` — per-scenario molecule tests.
- `CHANGELOG.md` — auto-generated by `release-please`; do NOT hand-edit.
- the version `galaxy.yml` is also updated via release-please action. NEVER update it manually.

## Conventions

- **Role naming:** `snake_case`, single-purpose, verb-first when it's an action (`dane_publish`, `lego_certificate`) or noun-first when it's a service (`postfix`, `haproxy`).
- **Variable prefixing:** every variable in a role's `defaults/main.yml` is prefixed with the role name (`postfix_*`, `haproxy_config_*`, `dane_*`). No bare names.
- **Examples in comments:** use IANA-reserved names (`example.com`, `192.0.2.x`, `2001:db8::x`). Never use real domains/IPs the caller might mistake for "the canonical default."
- **Comments explain *why*, not *what*.** A comment like `# enable HSTS` is noise; `# HSTS preload is irreversible — keep opt-in` earns its lines.
- **Idempotency:** every task must be re-runnable. Use `creates:`, handlers, and `changed_when` to mean it.
- **Notify-Handler**: if a task is meant to run as a result of another action, e.g. reload of haproxy service due to config change, it goes into handlers/main.yml.
  however, if a service needs to start or set enabled in a role, that'll be a task in tasks/\*.yml.

## Verification

Before any change lands:

1. `ansible-playbook --syntax-check playbook.yml`
2. Render the affected template(s) with the role's defaults and confirm output.
3. If touching a role that has `molecule/` scenarios, run them.
4. Spot-check the README's auto-generated variable table matches `defaults/main.yml`.

## When unsure

If you're tempted to add a caller-specific value, **stop**. Parameterise it as a role variable with a safe-or-empty default, document it in the README, and let the consumer set it.
